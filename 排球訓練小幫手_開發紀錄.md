## 一、專案現況總覽

一個給小朋友（國小，5 位內測）使用的 AI 排球訓練網頁 App，已到 **v2+**。
教練後台建名冊 → 小選手（家長陪同）填資料、選模式生成菜單、互動式跟練、打卡、上傳影片給 AI 分析。

### 核心流程
1. **登入分流**：👦 小選手（輸入名字）／🎓 教練後台（密碼，本機測試免密碼）
2. **教練建名冊**（只建名字的空殼帳號）
3. **小選手完善資料**（年級、年齡、性別、身高體重、家裡器材）
4. **選訓練模式生成菜單**：
   - 綜合訓練（含排球）：可複選多項技巧，含影片 AI 分析
   - 身體素質強化（不帶球）：純體能，不含影片分析
5. **互動式跟練 + 打卡 + 影片分析 + 進度／成長曲線**

### 資料庫（Supabase）
- 資料表：`students`、`training_logs`（含 `day_number`）、`body_metrics`、`weekly_journals`、`competitions`
- Migration 腳本（依序）：`setup_all.sql` → `migrate_v2.sql` → `migrate_v3.sql` → `migrate_v4.sql` → `migrate_v5.sql`
- Storage buckets：`volleyball-videos`（影片）、`journal-photos`（日誌照片）
- ⚠️ 部署/新環境要先在 Supabase SQL Editor 跑這些腳本，並建好兩個 public bucket + 政策

## 二、本次 session 完成的修改（依時間）

### 1. 🐛 修好「影片分析會崩潰」的關鍵 bug
- `save_curriculum()` 原本**沒把 `target_skill` 存進 DB**，導致影片分析時 `get_skill_checkpoints(None)` 丟 `TypeError`，側欄/標題也顯示不出技巧。
- 修法：存菜單時一併寫入 `target_skill`；`get_skill_checkpoints` 與影片呼叫端都加上 None 防呆。

### 2. ⚡ 強化 Gemini 重試 + 修正 429 沒被攔截的 bug
- 重試從 3 次 → 5 次，改成**指數退避 + 隨機抖動**。
- 原本只攔 `ServerError`，導致 **429（屬 ClientError）根本沒在重試**；改攔基底 `APIError` 並依錯誤碼判斷。
- 非過載錯誤（400/401/403）原樣拋出，不再誤判成「太忙」。

### 3. 🎛️ 重試狀態顯示到 UI
- `gemini_generate` 加 `status_cb` 回呼；生成菜單／影片分析的等待畫面改用 `st.status`，重試時顯示「⏳ AI 教練有點忙，正在多試幾次（第 N/5 次）…」。

### 4. 📋 菜單改條列式 + 🎮 互動式訓練播放器
- AI 菜單 schema 改成每個訓練日含 `items` 陣列，每項標明 **計次（reps×sets）** 或 **計時（seconds）**。
- 每天顯示「今天共 N 項」條列清單（份量徽章 🔁/⏱️ + 器材 + 提醒）。
- 純前端 JS 播放器「▶ 開始訓練」：計時項目自動倒數、計次項目點「完成一組」填圓點，頂部進度，全部完成顯示慶祝。
- **舊菜單（只有 task 字串）自動相容**。

### 5. 🔔 倒數提示音 + 震動
- 播放器用 Web Audio 合成「嗶」聲（時間到兩音、全部完成三連音）+ `navigator.vibrate`。
- 音訊在按「開始」時解鎖，**手機才能發聲**。

### 6. ✅ 做完 → 出現打卡鈕（使用者選的方案）
- 「📋 訓練菜單」中**今天那一天**（依星期判斷）的播放器下方有原生 Streamlit 打卡鈕。
- 與「✅ 今日打卡」分頁共用同一筆紀錄（`get_today_log`/`create_training_log`），**不會重複計分**。
- 原因：在 Streamlit Cloud 上，瀏覽器內嵌框無法可靠把「做完了」回傳伺服器，故用原生按鈕最穩。

### 7. 👥 教練後台：新增／編輯／刪除學員
- 左側欄學員名片下方：**✏️ 編輯資料**、**🗑️ 刪除學員（二次確認）**。

- 移除「🛠️ 教練工具」裡重複的編輯分頁，只留「🔄 重新生成菜單」。

### 8. 🐛 修好「新增學員沒反應」的 bug

- 修法：新增 `adding_student` 旗標；新增模式中下拉選單不自動搶回，並加「← 返回學員」。刪除時清掉 selectbox 記住的舊值避免報錯。

### 9. 🎯 技巧改可複選 + 影片選「練哪一項」

- AI 被要求把所選技巧全部分散排進菜單。
- 上傳影片時若有多項技巧 → 跳出 selectbox 選「這支影片是練哪一項」，分析鎖定該技巧的檢核點。

### 10. ✏️ 小選手也能回上一步改資料

### 14. 📔 每週訓練日誌（心情＋文字＋照片，強制關卡）
- 新表 `weekly_journals`（**需跑 `migrate_v4.sql`**）：student_id、plan_started_at、week_number、mood、content、photo_url、時間。
- 綁 `plan_started_at`＋週數 → 不同菜單的同週數不衝突，**累積成長歷史供未來匯出**。
- 照片存 Supabase Storage **`journal-photos`**（需手動建 public bucket，步驟見 `migrate_v4.sql` 註解）。
- 「📔 訓練日誌」分頁：一週一篇，心情表情（`JOURNAL_MOODS`）＋文字（必填）＋一張照片（選填），可改；教練唯讀。
- **強制關卡**：完成某週 5 天後，沒寫該週日誌就擋住下一週訓練（`render_journal_gate`，在 main 的 2.5 步）。教練不擋。
- 防呆：日誌表/bucket 未建好時，關卡略過、分頁顯示友善提示，不影響訓練。
- 🔜 未來：匯出成「成長日記」（圖＋文字，檔案格式待討論）；`get_all_journals` 已備好全量讀取。
- ⚠️ **必做設定**：跑 `migrate_v4.sql` ＋ 建 `journal-photos` public bucket。

### 13. ✅ 逐日獨立完成追蹤（每一天各自打卡）
- 問題：原本打卡用「今天日期＋週數」當鍵 → 今天打過一次卡，**每一天卡片都顯示「已完成」**（無法分辨第幾天）。
- 解法：`training_logs` 新增 **`day_number`** 欄位（**需跑 `migrate_v3.sql`**），每一天（第1天~第5天）各自記錄。
- 改動：
  - DB：`get_session_log` / `create_session_log`（取代 get_today_log / create_training_log）。
  - `completed_sessions` 改以 **(週,天) 去重**計數；`is_session_done` 判斷單天完成。
  - 菜單每天卡片：各自「✅ 完成第 N 天」按鈕，完成後標題顯示「✅ 已完成」；可同一天連做多天。
  - 「今日打卡」分頁 → 改為 **「✅ 今日完成」總覽**（列出今天完成哪幾天 + AI 回饋）。
  - 影片分析分頁：新增「這是第幾天」選擇，AI 分數/回饋掛到該天紀錄並標記完成。
- ⚠️ **部署前必跑**：Supabase SQL Editor 執行 `migrate_v3.sql`。

### 12. 🗓️ 進度改「自我配速」，不綁日曆（重要設計修正）
- 原本「第幾週」用 `plan_started_at` 的日曆天數算（每 7 天自動跳週）→ 會發生「禮拜五才開始卻被日曆推著走、做不完當週」。
- 改成 **`calculate_current_week` / `is_plan_finished` 用「完成打卡次數」推算**（每 5 次進一週，`DAYS_PER_WEEK=5`）。
- 影響：**禮拜幾開始都沒差**；跳過的日子可在週末或有空時補做；每個小朋友照自己進度走。
- 菜單日改標「**第 N 天**」（不再用週一～週五），休息日改標「😴 休息 / 補做日」，並加自我配速說明文字。
- 打卡鈕已不綁星期幾（見第 6 項修正），做完哪天就在那天打卡。
- ⚠️ 簡化點：目前未追蹤「一週內哪幾天做了」，只算總完成次數（小孩多半照 1→5 順序做，MVP 可接受）；若要嚴格逐日追蹤，需在 `training_logs` 加 `day_number`。

### 💳 關於 billing（使用者決定先不開）
- 使用者上次開 billing 曾不慎超額、退款流程久 → **決定留在免費層**。
- 免費層在尖峰會間歇性 503，靠重試/避開尖峰時段即可；10 人規模可接受。
- 若未來重新考慮付費，務必先設**硬性配額上限**（Google Cloud → APIs & Services → Generative Language API → Quotas，把每分鐘/每日請求數調到安全值），而非只設預算警示（警示不會自動停用），即可避免再次失控。

### 📦 Storage 容量規劃（使用者決定先保持現狀）
- 釐清：bucket 顯示的「50 MB」是**單一檔案上限**，不是總容量。
- 免費層總量：**Storage 1 GB**、**Database 500 MB**。
- 10 人估算：照片（~2MB/張）約撐一年；**影片（~10MB/段）約 2–3 個月會滿 → 影片是瓶頸**。
- **決定：先保持現狀**，等快滿了再處理。未來可選方案（由省到花錢）：
  1. **影片不永久保存**：分析完不備份，或自動清掉 N 週前舊影片（保留 AI 回饋文字）。← 最有效
  2. **照片上傳前壓縮**（縮到長邊 ~1280px，省約 10 倍；需加 Pillow）。
  3. 真的不夠才升 Supabase Pro（$25/月、100GB）。
- 兩個防護（影片自動清理 / 照片壓縮）尚未實作，待容量接近上限再做。

### A. 生成菜單前可指定「重點加強項目」（優先度高、可近期做）
- 需求：生成（或重生）菜單前，讓使用者勾選想重點加強的面向，使 AI 更貼合需求地分配內容。
- **確定的候選清單**（可複選；2026-06-24 與使用者討論後定案，**落地緩衝先排除**）：
  - **肌力類**：手部/握力、腿部肌力、旋轉力（核心轉體）、肩部肌力與穩定、核心穩定、手腕/前臂力量
  - **爆發/速度類**：彈跳力、爆發力、移動速度/短距衝刺、敏捷性
  - **控制/協調類**：協調性、反應速度、平衡感、節奏感/步法
  - **健康/防護類**：柔軟度/伸展、心肺耐力/肌耐力
  - **技術穩定類**（與現有 `target_skill` 球類技巧整合）：扣球穩定、發球、接發球、舉球/二傳、移位防守、攔網、低手接球（挖球）
  - 📌 給國小階段優先推：協調性、平衡感、肩部穩定、柔軟度（表現 + 防護兼顧）
  - ❌ 已排除：落地緩衝（本次不納入）
- 設計初步想法：
  - 在 `render_generate_menu` 加一個 `st.multiselect("想重點加強的地方")`，存成欄位（例如 `emphasis`）。
  - 把所選項目注入 `generate_curriculum` 的 prompt（類似現有 `target_skill` 多技巧的做法），要求 AI 把這些面向加重比例排進每日 `items`。
  - 與現有 `target_skill`（球類技巧）並存：`target_skill` 偏「練哪個技術動作」，`emphasis` 偏「想強化的身體素質/穩定度」。
- 規模評估：**小到中**，沿用既有 multiselect + prompt 機制即可；需在 DB students 加一欄（或併進現有 JSON）。
- ✅ **已實作（2026-06-24）**：常數 `EMPHASIS_OPTIONS`（依類別排序）；`render_generate_menu` 新增「💪 想重點加強的地方」複選欄；`generate_curriculum` 新增 `emphasis` 參數並注入 prompt（含國小生只用自體重/輕負荷的安全限制）；選項**存進菜單 JSON**（`curriculum["emphasis"]`，免 migration）。預設值由功能 D 的階段建議帶入。

### D. 賽事行事曆 + 賽前期程提醒（週期化建議）⭐
- 概念：依**距離下一場比賽的時間**安排訓練階段（運動科學的「週期化 Periodization」）。
- **設定範圍（已定案）**：教練後台維護**一份全隊共用**的賽事行事曆。
- **資料**：新表 `competitions`（全隊共用）→ 名稱、日期、（可選）重要程度（A級重點賽/B級一般賽，影響賽前調整期長度）；教練可新增/編輯/刪除。
- **互動深度（已定案，使用者選的）**：採「**生成菜單前先提醒 + 給建議**」，**不做 AI 全自動重排整段**。
  - 進入生成/重生菜單頁時，自動找**最近的未來賽事**、算出**距今幾週**，頁面頂部顯示提醒，例如：
    > 🏐 距離【XX盃】還有 **5 週**（2026-09-01）→ 目前屬「**專項強化期**」，建議重點：爆發力、彈跳、敏捷、專項技術。
  - 這個建議可（a）**自動預選 A 的「重點加強」項目**、（b）**一併注入 AI prompt**，讓生成的菜單貼合該階段。
- **階段對照表**（距比賽時間 → 階段 → 建議重點）：
  | 階段 | 距比賽 | 建議重點（對應 A 項目） | 量/強度 |
  |------|------|------|------|
  | 準備期/休賽季 | >6–8 週 或無賽事 | 大部位肌力、核心、爆發基礎、體能 | 量大、打基礎 |
  | 專項強化期 | 約 3–6 週 | 爆發力、彈跳、敏捷、專項技術 | 強度升、量略降 |
  | 賽前調整期(Taper) | 約 2 週 | 柔軟度/伸展、技術穩定(扣球/發球)、協調平衡、健康控制 | 大幅減量、避免痠痛疲勞 |
  | 賽後過渡期 | 賽後約 1 週 | 緩和、恢復、低強度 | 最輕、休息為主 |
- **與「自我配速」並存**：訓練進度仍照完成次數算（不綁日曆）；**階段建議則看真實日期**距比賽多近。兩者不衝突。
- ⚠️ **小朋友安全**：所謂「大部位重量/爆發」對國小生＝**自體重、輕負荷、動作品質優先**，prompt 內要限制年齡適切性，不扛槓鈴。
- 🔗 與 A 強整合：階段自動決定哪些「重點加強」當主角。
- ✅ **已實作（2026-06-24）**：
  - 新表 `competitions`（**需跑 `migrate_v5.sql`**）；DB 函式 `get_competitions`/`add_competition`/`delete_competition`（讀取失敗會友善降級回空清單，不影響其他功能）。
  - `compute_training_phase(competitions, today)`：依日期判定階段（賽前≤14天 / 專項≤42天 / 賽後過渡≤賽後7天 / 否則準備期或一般準備期），回傳建議重點與說明；已用假資料測過五種邊界皆正確。
  - `render_competition_manager()`：教練在生成菜單頁的展開區新增/刪除比賽。
  - `render_phase_reminder()`：生成菜單前顯示「距離 XX 盃還有 N 週 → 階段 + 建議重點」橫幅（所有人可見），建議重點**自動預選**到 A 的複選欄。
  - prompt 注入賽事週期說明（賽前調整期要求 AI 明顯減量）。
- ⚠️ **部署前必跑**：Supabase SQL Editor 執行 `migrate_v5.sql`。
